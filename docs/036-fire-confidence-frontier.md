# FIRE 达标前沿

## 1. 功能目标

FIRE 达标前沿在一份已完成的正式模拟上冻结计划参数、市场输入、收益风险模型、seed 与路径，然后只改变一个业务变量，寻找达到目标成功率的离散边界。达标条件是模拟成功率的 Wilson 95% 下界不低于用户设置的目标，不以点估计代替置信下界。

页面提供四个固定问题：

| 问题 | 候选中改变 | 始终冻结 | 返回的边界 |
| --- | --- | --- | --- |
| 不同退休年龄可承受多少支出 | 整数退休年龄、首年年度退休支出 | 当前资产、退休前储蓄、退休稳定收入、持仓与权重、模型、seed 和路径 | 每个年龄的最大达标年度支出 |
| 不同退休年龄至少需要储蓄多少 | 整数退休年龄、首年年度储蓄 | 当前资产、年度支出、退休稳定收入、持仓与权重、模型、seed 和路径 | 每个年龄的最低达标首年年度储蓄 |
| 按当前方案需要多少资产 | 起始总资产；持仓按源金额比例缩放，源总资产为零时按冻结目标权重缩放 | 退休年龄、储蓄、支出、退休稳定收入、资产构成、模型、seed 和路径 | 保持当前方案所需的最低当前资产 |
| 停止新增储蓄后需要多少资产 | 首年年度储蓄固定为零，再改变起始总资产 | 原退休年龄、支出、退休稳定收入、资产构成、模型、seed 和路径 | 停止新增储蓄后所需的最低当前资产 |

金额只在用户给定的 `min / max / step` 离散网格上搜索，不在档位之间插值，也不向搜索域外推算。找到边界时同时保存相邻的未达标证据；整个搜索域均达标或没有达标值时，结果明确标记端点结论。

## 2. 冻结来源与时间语义

前沿运行必须指定一条属于当前计划、状态为 `complete`、路径数不少于 1000 的正式模拟。创建前会重新校验：

- 当前计划配置哈希仍等于源模拟的配置哈希；估值日属于该配置哈希的一部分；
- 当前市场输入哈希仍等于源模拟的市场快照哈希；
- 模拟引擎版本、冻结输入哈希、路径索引及成功/失败汇总一致；
- 评估路径数不超过源模拟路径数和系统上限；
- 搜索档位数、评估次数和 path-month 成本不超过计算预算。

因此，时间推进导致估值日、计划配置或市场数据发生变化时，旧源模拟会以明确的 stale/market-changed 错误被拒绝，用户需要先生成新的正式模拟。即使规范化参数和 `input_hash` 与历史运行相同，一次新的运行请求也会重新计算；系统不会把“同一计划”或“相同输入哈希”当作复用历史业务结果的依据。

`input_hash` 仍是重要的审计字段。它覆盖算法版本、源 run ID、源引擎/配置/市场哈希、完整冻结快照哈希、问题类型、目标成功率、评估路径、年龄范围、金额搜索域和计算预算，用于证明任务实际计算了哪一份输入。

## 3. 运行与幂等契约

创建接口：

```text
POST /api/v1/plans/:plan_id/fire-frontier-runs
```

运行请求与网络重试是两个不同概念：

- 新的请求（新的 `Idempotency-Key`，或未提供 key）始终创建独立 task/run，返回 `202`、`reused:false`；
- 同一 `Idempotency-Key` 携带同一 canonical input 重试时返回原 task/run，返回 `200`、`reused:true`；
- 同一 `Idempotency-Key` 携带不同 canonical input 时返回 `409 idempotency_conflict`；
- task、frontier run 与幂等键在同一数据库事务中创建，不留下孤儿或半绑定记录。

active-task dedupe key 只收敛同一幂等请求的并发重试，不再按整个计划或 input hash 合并不同的业务运行。数据库的 `worker_task_idempotency_keys.task_id UNIQUE` 约束保持不变，不改变其他 worker 任务的幂等语义。

## 4. 结果解释与图表

每个结果页必须显示“这次到底计算了什么”，包含：

- 本问题的作用、候选中改变的字段、始终冻结的字段和搜索逻辑；
- 来源模拟、当前/退休/终止年龄、源当前资产和年度现金流；
- 持仓缩放口径、收益风险模型、seed、路径数、金额网格；
- 通胀、提款、再平衡规则和 Wilson 达标门槛；
- 每个边界点的点估计、Wilson 95% 区间、路径数、配对改善/回退和相邻点证据。

年龄曲线明确标注横轴为退休年龄（岁）。“可承受支出”纵轴为最大达标年度退休支出（元）；“最低储蓄”纵轴为最低达标首年年度储蓄（元）。纵轴刻度按金额量级缩写并保留完整轴语义，避免大金额与轴标题重叠。鼠标悬浮或键盘聚焦坐标点时，展示年龄、金额、边界状态、成功路径数、成功率点估计、Wilson 95% 区间、目标判定和相邻未达标档位；详情浮层纵向约束在图内安全区域，在 375px 视口不产生横向滚动或越界。没有达标值的空心点不参与连线；曲线只连接离散可行点，不表示插值结果。

## 5. 错误可诊断性

可预期的业务错误使用稳定错误码和适当的 4xx 状态，例如 `frontier_source_stale`、`frontier_source_market_changed`、`frontier_budget_exceeded` 与 `idempotency_conflict`。前端错误提示同时展示服务端消息、错误代码和 `request_id`，便于用户将页面错误与服务端结构化日志关联。未知内部错误不向浏览器泄漏数据库语句或堆栈。

## 6. 验证不变量

自动化测试必须覆盖：

- 四类问题均能完成 readiness、创建、worker 执行、详情与点证据流程；
- 相同输入使用不同幂等键会产生不同 task/run；
- 同 key、同输入返回原任务，同 key、不同输入固定冲突；
- stale 配置、市场变化、无效源、预算边界在创建 task 前被拒绝；
- 目标成功率默认 95%，源路径足够时评估路径默认 10000；
- 年龄图轴含义和点 hover 详情可访问；
- 数据库 `integrity_check` 为 `ok` 且 `foreign_key_check` 为空。
