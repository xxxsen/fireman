## 3. P1 问题

### P1-1 `场景配置` 只管理大类权重，未覆盖模板内地区参数，导致“模板只读边界”仍未真正闭环

性质：实现缺失。

定位：

- `web/app/scenarios/page.tsx:27-49`
- `web/app/scenarios/page.tsx:78-82`
- `web/app/scenarios/page.tsx:178-189`
- `web/components/plans/AllocationSettings.tsx:54-55`
- `web/components/plans/AllocationSettings.tsx:105-108`
- `web/lib/api/allocation.ts:12-29`
- `web/types/api.ts:86-94`

问题：

`td/024` 的核心边界是：

1. `计划设置` 中的“当前计划目标配置”只能切换模板、只读查看模板参数；
2. 用户如需修改“场景内的参数”，只能去 `场景配置`。

但当前实现里，`场景配置` 页面和相关 API 只管理：

- `AllocationScenario.weights`，即大类权重

并不管理地区组内目标。与此同时，`当前计划目标配置` 页在切换模板时：

- 大类权重预览来自 `previewScenario?.weights`
- 地区组内权重继续直接读取当前计划的 `allocation.data?.region_targets`

代码里甚至显式提示：

> `地区组内权重沿用当前计划配置，保存场景切换后生效。`

这与 `td/024` 的定案是直接冲突的。结果是：

1. 用户在 `计划设置` 看到的“当前计划目标配置”并不是一个完整的模板视图；
2. 用户也无法在 `场景配置` 中修改地区参数，因为该入口根本不提供这部分编辑能力；
3. 最终形成“模板负责大类、计划私有数据负责地区”的混合模型，边界没有真正收口。

这不是文案问题，而是产品模型与实现模型不一致。

修复方案：

把“场景模板”提升为完整的目标配置对象，统一包含：

1. 大类权重；
2. 各大类下的地区组内权重。

相应调整：

1. `AllocationScenario` 类型与 `/allocation-scenarios` API 契约扩展为同时返回 / 保存 `region_targets`；
2. `场景配置` 页新增地区组内权重编辑；
3. `当前计划目标配置` 页在切换模板时，预览的大类和地区都来自同一个模板对象；
4. `applyScenario` 落地时同时应用模板中的大类与地区目标；
5. 删除“地区组内权重沿用当前计划配置”的临时逻辑与提示。

验收逻辑：

1. 在 `场景配置` 中可以完整编辑某模板的大类与地区参数。
2. 在 `计划设置 -> 当前计划目标配置` 中切换模板后，看到的大类与地区都来自该模板，而不是当前计划旧值。
3. 保存模板切换后，`getAllocation(planId)` 返回的大类与地区目标同时切换。
4. 用户不再需要在计划页和模板页之间猜测“哪部分属于模板，哪部分属于计划私有配置”。

### P1-2 `资产变更` 仍无法编辑组内配比，且会强制平均分配，违背“更新标的、配比等功能”的目标

性质：实现缺失。

定位：

- 原 `td/024` 方案中关于“资产变更承担更新标的、配比等功能”的条目（现已归档至 `docs/008`）
- 原 `td/024` 方案中关于“资产变更承担原持仓管理职责收拢”的条目（现已归档至 `docs/008`）
- `web/app/plans/[id]/asset-refresh/page.tsx:425-468`
- `web/lib/asset-refresh.ts:54-75`
- `web/lib/asset-refresh.ts:77-84`

问题：

`td/024` 与用户原始诉求都明确要求：

- `资产变更` 不只是改金额，还用于更新当前计划的持仓标的、配比等功能；
- 该流程应当承担原“持仓管理”的核心职责收拢。

但当前实现中，`录入当前资产` 页的表格只提供：

1. 启用/禁用；
2. 标的增删；
3. 当前金额输入。

它**没有提供任何 `weight_within_group` 的编辑入口**。更严重的是，当前代码在以下操作后会自动重算并平均分配组内权重：

- 启用/停用；
- 删除标的；
- 新增标的。

也就是说，用户不仅不能显式编辑组内配比，系统还会在结构变动时悄悄把组内配比改成平均值。这与“用于更新持仓标的、配比等功能”的目标相反，会产生真实业务偏差。

修复方案：

把 `资产变更` 的资产录入表扩展为同时编辑：

1. `current_amount_minor`
2. `weight_within_group`

并取消当前的“结构变化即强制平均分配”策略，改为：

1. 新增资产时，为新资产给出默认配比（可为 0 或按剩余比例初始化），但必须让用户可见、可改；
2. 启停 / 新增 / 删除后，保留现有组内配比并显式暴露校验结果；
3. 提交前对每个 `asset_class + region` 分组做 `weight_within_group` 合计校验；
4. 只有分组校验通过时才允许提交结构变更。

验收逻辑：

1. `资产变更 -> 录入当前资产` 页面中，每个资产行都能显式编辑组内配比。
2. 新增 / 删除 / 启停资产后，系统不会再静默平均重写用户的组内配比。
3. 每个组内配比不满足 100% 时，页面明确报错并阻止提交。
4. 提交后的 `updateHoldings` 请求中，`weight_within_group` 与用户实际输入一致，而不是系统自动平均值。
