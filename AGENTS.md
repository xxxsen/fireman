# agents.md

## 1. 项目定位

Fireman 是一个**单用户、本地优先**的 FIRE 资金配置、调仓与模拟系统，核心链路包括：

1. 资产资料库录入与市场数据清洗
2. 建立计划、维护三层权重
3. 对比当前持仓与目标配置，生成调仓建议或调仓草稿
4. 运行 FIRE 模拟、压力测试与敏感性分析

技术栈：

- 后端：Go、Gin、SQLite
- 前端：Next.js、React、TypeScript、Tailwind CSS
- 市场数据 sidecar：Python、FastAPI、AKShare、uv
- 编排：Docker Compose

---

## 2. 目录职责

仓库主要目录及其语义如下：

- `cmd/fireman/`：Go 程序入口
- `internal/`：后端业务代码，包含 API、service、repository、domain、simulation、jobs 等模块
- `migrations/`：SQLite schema 与增量迁移
- `web/`：前端应用与前端测试
- `sidecars/market-provider/`：市场数据 sidecar 及其 pytest 测试
- `docs/`：已完整实施、可长期维护的正式文档
- `td/`：临时方案、实施 review、过程文档
- `scripts/`：本地开发或 smoke 脚本
- `docker/`：Compose 与镜像相关配置

规则：

- 正式能力说明、接口说明、用户流程、实施结果，最终都应进入 `docs/`
- `td/` 只用于临时设计和实施过程记录，不是长期稳定文档
- 修改功能时，若发现 `docs/` 已过时，应一并修正

---

## 3. 文档规则

### 3.1 `td/` 与 `docs/` 的边界

- `td/` 中的方案一旦**完整实施并稳定**，应整理为自洽文档移入 `docs/`，随后删除对应的临时方案和 review 记录；`td/` 只保留尚未实施或明确暂缓的方案。
- 清理或重排 `td/` 编号前，必须先移除代码与正式文档对临时编号的引用；`docs/` 和代码应使用自包含的业务描述。
- 移入 `docs/` 的文档必须能独立阅读，不能依赖“查看某个本地临时方案”才能理解
- 实施结果或者 `docs/` 目录中的文档禁止提及或者引用某个 `td/` 目录中的文档


### 3.2 文档更新原则

当功能发生以下变化时，应同步更新文档：

- 用户主路径变化
- API 契约变化
- 关键计算公式变化

正式文档只记录稳定的功能行为、底层架构、接口契约、验证不变量和验证命令。不得写入某次执行产生的测试文件数、用例数、PASS 输出、临时任务 ID、现场数据库记录或待签字状态。测试章节描述“覆盖什么、必须满足什么”，不记录“本次通过了多少”。

## 4. 代码质量底线

未经用户明确许可，Agent 不允许通过降低标准来“让检查通过”。禁止行为包括：

- 修改 lint 配置以放宽规则
- 删除测试或降低测试强度来规避失败
- 用跳过检查、忽略错误、静默吞错的方式掩盖真实问题
- 为了绕过类型、lint、测试而加入无必要的 disable 注释
- 在功能未闭环时宣称“已完成”

优先策略应是：

1. 修正实现
2. 重构代码
3. 补充或修正测试
4. 更新过期文档

---

## 5. 代码修改原则

### 5.1 后端

- 业务语义优先放在 `internal/service/` 与 `internal/domain/`
- 数据访问逻辑放在 `internal/repository/`
- 避免把业务规则散落在 handler、SQL 拼接和前端中
- 涉及金额时统一使用 `minor` 单位整数，避免浮点金额持久化
- 计划、调仓、模拟相关改动要特别注意 `config_version`、事务边界和审计事件

### 5.2 前端

- 页面级逻辑放在 `web/app/`
- 可复用组件放在 `web/components/`
- API 调用、数据类型与纯计算逻辑分别放在 `web/lib/`、`web/types/`
- 纯计算逻辑优先抽到 `web/lib/` 并配套单测，不把复杂公式直接埋在组件 JSX 中

### 5.3 数据与迁移

- 已发布语义一旦入库，修改时优先新增 migration，而不是假设可重建数据库
- 迁移需考虑旧数据升级路径，不能只满足新建库场景
- 业务 schema 的 DDL（`CREATE TABLE` / `ALTER TABLE` / `CREATE INDEX` / `DROP` 等）只能出现在 `migrations/` 或测试 fixture 中；运行时代码不得做 schema repair / 动态 `ALTER TABLE`。需要变更已存在 schema 时，只新增顺序迁移文件（如 `migrations/0014_xxx.sql`），不要在 Go 运行时代码里写 DDL 修补

## 6. 测试纪律

### 6.1 必跑检查

如果任务涉及代码修改，完成后至少应按改动范围运行相应检查：

后端改动：

```bash
make build
make test
make install-golangci-lint
make lint-go
```

前端改动：

```bash
make web-install
make web-lint
make web-test
make web-build
```

sidecar 改动：

```bash
make market-provider-install
make market-provider-test
```

全链路改动或不确定影响范围时：

```bash
make ci
```

涉及后端接口、迁移、计划/调仓/模拟主链路时，还应运行：

```bash
make integration-test
```

以上检查是交付门禁，不只是建议命令。任务涉及代码修改时，所需的 build、test 和 lint 必须全部成功后才能宣称完成；lint 的 error 或 warning 均不得以“既有问题”为由留到后续。若失败来自当前授权范围之外且无法安全修复，应停止交付、明确报告阻塞项并等待用户决定，不能把失败的检查描述为通过。

### 6.2 测试编写要求

新增或修改代码时，测试应覆盖至少三类场景：

- 正常路径
- 错误路径
- 边界条件

优先级：

- 纯计算逻辑：单元测试
- API 契约与跨层流程：集成测试
- sidecar 适配器与 envelope：pytest

### 6.3 外部依赖约束

- 常规测试不得依赖真实外网或真实 AKShare 可用性
- sidecar 测试应优先使用 mock / fake 数据
- 不要把“人工 smoke”写成默认 CI 前提

## 8. Review 标准

当任务涉及代码修改，不应只看“命令是否通过”，还要主动检查：

- 是否引入回归
- 是否破坏既有口径
- 是否遗漏边界情况
- 是否存在文档与实现不一致
- 是否有未覆盖的关键测试缺口

发现问题时，优先级按以下口径理解：

- `P0`：崩溃、数据损坏、安全问题、不可恢复的错误写入
- `P1`：功能错误、计算错误、主链路失败
- `P2`：明显逻辑缺陷、边界遗漏、可预见的维护风险
- `P3`：风格、命名、结构可读性问题

代码完成的最低标准是：没有遗留明确的 `P0` / `P1` / `P2` / `P3` 级问题。

---

## 9. 交付要求

完成任务时，Agent 应明确说明：

- 改了什么
- 为什么这样改
- 跑了哪些检查
- 哪些检查没有跑，以及原因
- 是否涉及文档同步

如果只是做文档归档或方案整理，也应保证：

- 文档内容完整、自洽
- 不依赖仓库外部路径
- 不要求读者再去查其它临时文件才能理解

---

## 10. 本仓库的默认工作方式

除非用户另有要求，Agent 在本仓库中应默认采取以下工作方式：

1. 先读代码与文档，再下结论
2. 先核对仓库真实状态，再写规范或说明
3. 尽量让实现、测试、文档三者同步
4. 对已完整实施的临时方案，及时归档到 `docs/`
5. 不把外部项目、外部路径或第三方私有文件作为最终交付的一部分

这份 `agents.md` 本身也遵循上述规则：它应能单独说明 Fireman 仓库内 Agent 的行为边界，而无需依赖任何其它仓库文件。
